## 프로젝트 배경

보안 운영에서 중요한 것은 이벤트가 발생했다는 사실 자체보다, 그 사실을 얼마나 빨리 알아차리고 바로 해석 가능한 형태로 전달하느냐입니다. 탐지는 되어도 사람이 늦게 보거나, 알림이 단순 텍스트로 와서 다시 원문 로그를 찾아야 한다면 실제 대응 속도는 크게 좋아지지 않습니다.

특히 방화벽 이벤트는 발생량이 많고 맥락이 복잡해서, 단순 알림만으로는 현업에서 쓰기 어렵습니다. 어떤 룰에 걸렸는지, 어떤 장비에서 왔는지, 얼마나 빨리 봐야 하는지까지 같이 보여줘야 운영자가 바로 판단할 수 있습니다.

이 프로젝트는 FortiGate 이벤트를 Splunk에서 탐지하고, 그 결과를 Slack Block Kit 형태로 구조화해 전달하는 체계를 만들기 위해 시작했습니다. 핵심 목표는 탐지 품질과 알림 전달 속도를 함께 높여, 이벤트 발생 후 30초 이내에 운영자가 상황을 파악할 수 있게 하는 것이었습니다.

## 문제 상황

- 보안 이벤트는 발생하지만 운영자가 즉시 확인하기 어려운 경우가 있었습니다.
- 단순 텍스트 알림은 정보 해석 비용이 높아 대응 우선순위 판단이 느렸습니다.
- 방화벽 로그를 계속 직접 확인하는 방식은 피로도가 높고 누락 가능성도 있었습니다.
- 탐지 규칙과 알림 체계가 분리돼 있어, 탐지 이후 전달 단계에서 지연이 생길 수 있었습니다.
- 결과적으로 32개 탐지 룰을 운영하면서도 이벤트 발생 후 30초 이내에 바로 이해 가능한 알림 체계가 필요했습니다.

## 해결 접근

1. FortiGate syslog를 Splunk로 수집해 탐지 기준을 일관되게 만들었습니다.
2. 32개 탐지 룰을 운영 기준에 맞춰 정리했습니다.
3. 탐지 결과를 Python으로 가공해 Slack Block Kit 형태로 보냈습니다.
4. 단순 발송보다, 운영자가 바로 판단할 수 있는 알림 구조를 만드는 데 집중했습니다.
5. 이벤트 발생부터 알림 도달까지 짧은 경로를 유지하도록 파이프라인을 단순화했습니다.

### 1. 수집과 탐지를 먼저 안정화

알림 시스템은 메시지를 잘 보내는 것보다 먼저, 어떤 이벤트를 신뢰할 수 있게 잡아내는지가 중요합니다.

그래서 FortiGate syslog를 Splunk로 수집하는 흐름을 기준점으로 잡고, 탐지 규칙 운영이 가능한 상태를 먼저 만들었습니다.

### 2. 탐지 룰을 운영 가능한 범위로 정리

룰이 많다고 좋은 것은 아닙니다. 너무 적으면 놓치고, 너무 많으면 운영자가 지칩니다.

이 프로젝트에서는 32개 탐지 룰을 운영하면서, 실제로 계속 확인 가능한 수준의 탐지 체계를 유지하는 데 초점을 맞췄습니다.

### 3. Python으로 알림 포맷 가공

Splunk에서 탐지했다고 해서 곧바로 좋은 알림이 되는 것은 아닙니다. 운영자가 바로 읽을 수 있는 형태로 정보를 가공하는 과정이 필요했습니다.

Python을 사용해 탐지 결과를 정리하고, Slack Block Kit 포맷으로 구성해 핵심 정보가 한눈에 들어오도록 만들었습니다.

### 4. 전달 채널은 Slack Block Kit로 표준화

알림 채널은 결국 사람이 소비하는 접점입니다. 그래서 예쁘게 보이는 것보다, 빠르게 읽히고 중요한 정보가 먼저 보이도록 구조화하는 것이 중요했습니다.

Slack Block Kit를 사용한 이유도 여기에 있습니다. 제목, 이벤트 맥락, 확인 포인트를 구분해 보여주기 좋았습니다.

### 아키텍처 개요

- FortiGate syslog가 이벤트 원천 역할 수행
- Splunk가 수집 및 탐지 허브 역할 수행
- Python이 탐지 결과를 가공하고 알림 메시지를 생성
- Slack Block Kit가 운영자 전달 채널 역할 수행

### 구현 포인트 1. 수집, 탐지, 전달 흐름 분리

한 시스템 안에서 모두 처리하려고 하면 변경과 운영이 어려워집니다. 그래서 각 단계의 역할을 나눴습니다.

- FortiGate는 이벤트 발생 원천
- Splunk는 수집과 탐지
- Python은 메시지 가공
- Slack은 전달

이 구조는 장애 원인 파악과 유지보수 모두에 유리했습니다.

### 구현 포인트 2. 운영자가 바로 읽는 메시지 설계

보안 알림은 길고 상세한 원문보다, 즉시 판단 가능한 구조가 중요합니다. 그래서 알림 내용을 Block Kit 기준으로 정리해 정보 우선순위를 드러냈습니다.

이 덕분에 운영자는 메시지를 보는 순간 무엇을 먼저 확인해야 하는지 빠르게 파악할 수 있었습니다.

### 구현 포인트 3. 30초 내 알림 도달 목표

이 프로젝트의 성패는 예쁜 UI가 아니라 속도에 달려 있었습니다. 이벤트 발생 후 30초 이내 알림이라는 목표를 두고, 파이프라인 경로를 불필요하게 길게 만들지 않는 데 집중했습니다.

### 구현 포인트 4. 탐지 규칙 운영성과 확장성 균형

32개 탐지 룰은 적지 않은 규모입니다. 따라서 새 룰을 추가하는 것만큼, 기존 룰이 운영 가능한 상태를 유지하는 것도 중요했습니다. 이 프로젝트는 단기 실험이 아니라 실제 운영 체계를 만드는 방향으로 진행했습니다.

## 결과

- 탐지 운영 범위: 단편적 이벤트 확인 → **32개 탐지 룰 운영**
- 전달 채널: 단순 메시지 중심 → **Slack Block Kit 기반 구조화된 알림**
- 수집 흐름: 장비 개별 확인 → **FortiGate syslog → Splunk 파이프라인 구성**
- 대응 속도: 수동 확인 의존 → **이벤트 발생 후 30초 이내 알림 도달**

정량 지표 외에도 운영 효율이 개선됐습니다. 이벤트를 발견한 뒤 다시 내용을 정리해 전달하는 수작업이 줄었고, 운영자는 먼저 봐야 할 이벤트를 빠르게 판단할 수 있게 됐습니다.

또 이 구조는 이후 다른 보안 자동화와 연결하기 쉬운 기반이 됐습니다. 탐지와 전달이 정리되면 후속 대응 자동화도 설계하기 쉬워집니다.

## 사용 기술 상세

### Splunk

보안 이벤트 수집과 탐지 중심 도구로 사용했습니다. 탐지 체계를 운영 가능한 상태로 유지하는 데 핵심 역할을 했습니다.

### FortiGate syslog

보안 이벤트의 원천 로그로 활용했습니다. 실시간 탐지 체계의 입력 데이터 역할을 담당했습니다.

### Python

탐지 결과를 사람이 읽기 쉬운 알림 메시지로 가공하는 자동화 계층으로 사용했습니다.

### Slack Block Kit

운영자 전달 채널로 사용했습니다. 알림 정보를 구조화해 즉시 읽히는 형태로 정리하는 데 적합했습니다.
