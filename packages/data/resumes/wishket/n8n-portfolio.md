## 프로젝트 배경

운영 업무에는 생각보다 반복이 많습니다. 알림 전송, 상태 점검, 외부 서비스 연동, 데이터 수집 같은 일은 한 번만 할 때는 단순하지만, 매일 반복되기 시작하면 사람의 시간을 계속 소모합니다.

특히 여러 시스템을 함께 운영하는 환경에서는 작은 수동 작업이 누적돼 전체 생산성을 떨어뜨립니다. 같은 내용을 여러 서비스에 입력하거나, 상태를 확인한 뒤 다른 채널로 다시 전달하는 일은 자동화가 가능하지만, 실제로는 누군가의 기억과 습관에 의존하는 경우가 많습니다.

이 프로젝트는 그런 운영 절차를 사람 중심에서 워크플로우 중심으로 바꾸기 위해 시작했습니다. 목표는 단순한 편의 기능이 아니라, 반복 업무를 줄이고 실행 누락 가능성을 낮추며, 필요한 연동을 빠르게 확장할 수 있는 자동화 기반을 만드는 것이었습니다.

## 문제 상황

- 알림, 배포, 데이터 수집 같은 반복 운영 업무가 사람 손에 의존하고 있었습니다.
- 여러 서비스 사이를 오가며 같은 내용을 다시 입력하거나 전달하는 비효율이 있었습니다.
- 단일 작업은 단순하지만 전체 흐름은 길어서 누락과 지연이 발생하기 쉬웠습니다.
- 자동화가 필요한 업무가 쌓이는데도 표준화된 워크플로우 레이어가 없었습니다.
- 결과적으로 15+ 워크플로우를 안정적으로 운영하고, 일 200+ 실행을 감당할 수 있는 구조가 필요했습니다.

## 해결 접근

1. 자주 반복되는 운영 절차를 먼저 추려서 워크플로우 단위로 분리했습니다.
2. n8n을 중심으로 실행 흐름을 표준화하고, PostgreSQL과 Docker로 운영 기반을 정리했습니다.
3. Slack, GitHub, Cloudflare, Grafana처럼 실제 운영에서 자주 쓰는 서비스부터 연동했습니다.
4. 사람이 확인만 하면 되는 단계와, 기계가 자동으로 처리할 수 있는 단계를 분리했습니다.
5. 개별 자동화가 아니라 재사용 가능한 운영 패턴을 만들도록 설계했습니다.

### 1. 반복 업무를 워크플로우 후보로 정의

자동화는 기술보다 대상 선정이 먼저입니다. 매일 또는 자주 반복되면서, 실수 비용이 있고, 서비스 간 전달 과정이 긴 작업을 우선 대상으로 잡았습니다.

이 기준으로 보면 알림, 배포, 데이터 수집은 자동화 효과가 큰 영역이었습니다.

### 2. n8n을 운영 오케스트레이션 레이어로 선택

n8n을 선택한 이유는 복잡한 운영 절차를 시각적인 흐름으로 정리하면서도, 여러 외부 서비스를 빠르게 연결할 수 있기 때문입니다.

단순 스크립트 중심 접근은 처음엔 빠르지만, 시간이 갈수록 흐름 추적과 유지보수가 어려워집니다. 반면 n8n은 누가 봐도 현재 자동화가 어떻게 움직이는지 이해하기 쉽고, 변경 범위도 파악하기 좋습니다.

### 3. 운영 데이터는 PostgreSQL에 정리

워크플로우가 늘어나면 실행 이력과 상태 관리가 중요해집니다. PostgreSQL을 기반으로 운영 데이터를 안정적으로 관리해 워크플로우 수가 늘어도 운영 기준이 흔들리지 않게 했습니다.

### 4. Docker로 실행 환경 일관성 확보

자동화 플랫폼은 한 번 만들어 두고 끝나는 시스템이 아니라 계속 수정되고 늘어나는 운영 자산입니다. Docker를 사용해 실행 환경을 일관되게 유지하고 재배포 부담을 낮췄습니다.

### 5. 통합보다 연결 비용 절감에 초점

이 프로젝트에서 중요한 것은 많은 시스템을 붙였다는 사실이 아니라, Slack, GitHub, Cloudflare, Grafana를 하나의 운영 흐름 안에서 이어 붙였다는 점입니다.

운영자는 그 결과로 여러 페이지를 오가며 수동 전환할 필요가 줄어들었습니다.

## 구현 상세

이 프로젝트의 구현 핵심은 개별 자동화 스크립트를 모아두는 것이 아니라, 운영 작업을 워크플로우 자산으로 전환하는 데 있었습니다.

### 아키텍처 개요

- n8n이 자동화 흐름 실행을 담당
- PostgreSQL이 워크플로우 운영 데이터 기반 역할 수행
- Docker가 배포 및 실행 환경 표준화 담당
- Slack, GitHub, Cloudflare, Grafana가 주요 외부 연동 지점으로 연결

### 구현 포인트 1. 워크플로우를 목적별로 분류

자동화는 많아질수록 관리가 중요해집니다. 그래서 워크플로우를 기능 기준이 아니라 운영 목적 기준으로 나눴습니다.

- 알림 워크플로우
- 배포 보조 워크플로우
- 데이터 수집 워크플로우

이 구조는 15+ 워크플로우를 운영하면서도 확장 방향을 잃지 않게 해줬습니다.

### 구현 포인트 2. 외부 서비스 연동 단순화

운영자는 업무를 끝내기 위해 GitHub, Cloudflare, Grafana, Slack을 각각 따로 열어보는 경우가 많습니다. n8n으로 연동한 뒤에는 필요한 정보를 한 흐름 안에서 이동시키는 구조를 만들 수 있었습니다.

이 변화는 작업 속도보다도 누락 방지 측면에서 효과가 컸습니다.

### 구현 포인트 3. 사람이 개입할 위치를 명확히 구분

모든 단계를 완전 자동화하는 것이 늘 최선은 아닙니다. 확인이 필요한 단계는 남기고, 반복 전달과 상태 연결처럼 자동화 효과가 큰 단계부터 기계가 맡도록 정리했습니다.

이 방식은 운영 통제력을 유지하면서도 수동 작업을 줄이는 데 적합했습니다.

### 구현 포인트 4. 실행량을 감당하는 운영 구조

일 200+ 실행은 단순 체험 수준이 아니라 실제 운영에 가까운 빈도입니다. 따라서 개별 워크플로우 작성보다, 여러 흐름이 동시에 돌아가도 관리 가능한 기준을 만드는 데 더 신경 썼습니다.

## 결과

- 자동화 범위: 일부 스크립트 중심 → **15+ 워크플로우 운영**
- 실행 규모: 비정형 수동 실행 → **일 200+ 실행**
- 통합 범위: 개별 서비스별 작업 → **Slack, GitHub, Cloudflare, Grafana 연동**
- 운영 방식: 사람 중심 전달 → **워크플로우 중심 반복 업무 자동화**

정량 지표 외에도 의미 있는 변화가 있었습니다. 자동화 흐름이 시각적으로 보이기 시작하면서, 운영 절차를 다른 사람에게 설명하거나 개선 포인트를 찾는 비용이 줄었습니다.

또 알림, 배포, 데이터 수집처럼 성격이 다른 작업을 같은 플랫폼에서 다루게 되면서 자동화의 재사용성이 높아졌습니다.

## 사용 기술 상세

### n8n

자동화 오케스트레이션의 중심입니다. 반복 업무를 시각적 워크플로우로 정리하고, 연동 구조를 빠르게 추가하는 데 사용했습니다.

### PostgreSQL

워크플로우 운영 데이터의 기반으로 사용했습니다. 실행량이 늘어나도 관리 기준이 흔들리지 않도록 뒷단을 안정적으로 유지하는 역할을 맡았습니다.

### Docker

실행 환경 표준화를 위해 사용했습니다. 운영 환경 재현성과 배포 일관성을 확보하는 데 도움이 됐습니다.

### Slack

운영자 알림과 상태 전달 채널로 활용했습니다. 자동화 결과를 가장 빠르게 소비하는 접점입니다.

### GitHub

코드 및 작업 흐름과 연동되는 지점으로 사용했습니다. 개발과 운영이 분리되지 않도록 연결하는 데 의미가 있었습니다.

### Cloudflare

배포나 운영 작업과 이어지는 외부 서비스 연동 대상 중 하나로 사용했습니다.

### Grafana

운영 지표나 상태를 자동화 흐름과 이어서 확인하는 연동 대상으로 사용했습니다.
