## 프로젝트 배경

여러 서비스와 인프라를 직접 운영하다 보니, 장애를 감으로 추적하는 방식의 한계가 빠르게 드러났습니다. 개별 서비스 상태는 보이더라도, 메트릭과 로그를 한 자리에서 연결해 보지 못하면 원인 파악이 늦어지고 같은 문제를 반복해서 확인해야 했습니다.

특히 홈랩 환경은 규모가 작아 보여도 관측 포인트가 흩어지기 쉽습니다. 서비스가 늘어날수록 메트릭 수집 대상, 로그 확인 위치, 외부 엔드포인트 상태 확인 방식이 분산되기 때문에 운영자는 매번 여러 도구를 오가야 합니다. 이 프로젝트는 그런 분산된 관측 작업을 하나의 운영 흐름으로 묶기 위해 시작했습니다.

단순히 대시보드를 예쁘게 만드는 것이 목적은 아니었습니다. 운영자가 문제를 더 빨리 발견하고, 원인을 더 짧은 경로로 좁히고, 장기적으로는 안정적인 가용성을 유지할 수 있는 기반을 만드는 것이 핵심 목표였습니다.

## 문제 상황

- 서비스 상태 확인 기준이 도구별로 분산되어 있어 운영 맥락이 끊기는 문제가 있었습니다.
- 메트릭과 로그를 따로 봐야 해서 이상 징후를 발견해도 원인 파악까지 시간이 더 걸렸습니다.
- 외부 엔드포인트 가용성 감시가 일관되지 않아 장애 징후를 선제적으로 보기 어려웠습니다.
- 운영 대상이 늘면서 수집 포인트 관리가 반복 업무로 바뀌고 있었습니다.
- 단순 상태 확인을 넘어 추세와 상관관계를 같이 보는 화면이 필요했습니다.
- 결과적으로 50+ 메트릭 엔드포인트, 3개 데이터소스, 12+ 대시보드를 하나의 운영 체계로 정리할 필요가 있었습니다.

## 해결 접근

1. 메트릭, 로그, 상태 확인을 분리하지 않고 같은 운영 흐름 안에서 보이도록 설계했습니다.
2. Prometheus, Loki, Elasticsearch를 데이터 성격에 맞게 나누고 Grafana에서 통합 조회하게 구성했습니다.
3. Blackbox Exporter를 포함해 외부 엔드포인트 상태까지 관측 범위에 넣었습니다.
4. 운영자가 자주 확인하는 질문을 기준으로 대시보드 구조를 나눴습니다.
5. 특정 도구 자랑보다 장애 감지와 원인 추적 속도를 높이는 방향으로 화면과 수집 체계를 정리했습니다.

### 1. 관측 대상을 먼저 구조화

처음부터 화면을 만들기보다, 어떤 신호를 반드시 놓치면 안 되는지부터 정리했습니다.

핵심은 세 가지였습니다.

- 메트릭은 Prometheus 중심으로 수집
- 로그는 Loki 중심으로 집계
- 추가 조회가 필요한 데이터는 Elasticsearch 연동

이렇게 나누면 각 도구의 역할이 분명해지고, 이후 Grafana에서 한 화면으로 묶었을 때도 왜 이 데이터가 여기 있는지 설명이 쉬워집니다.

### 2. 데이터소스별 역할 분리

Prometheus는 수치 기반 상태 확인에 적합하고, Loki는 로그 흐름 추적에 유리하며, Elasticsearch는 별도 검색과 집계 용도로 활용하기 좋습니다.

기술 선택 기준은 단순했습니다.

- 빠르게 상태를 보고 싶을 때는 메트릭
- 이상 징후의 문맥을 보고 싶을 때는 로그
- 추가 분석과 검색이 필요할 때는 Elasticsearch

이 분리는 운영자가 데이터를 찾는 시간을 줄이는 데 직접적으로 도움이 됐습니다.

### 3. Blackbox Exporter로 외부 가용성 감시 보강

내부 자원 상태만 좋다고 서비스가 정상인 것은 아닙니다. 실제로는 외부에서 접근 가능한지, 응답 가능한지 확인해야 운영 체계가 완성됩니다.

그래서 Blackbox Exporter를 포함해 외부 엔드포인트 관점의 상태 확인을 넣었습니다. 덕분에 내부 리소스와 외부 접근성 사이의 간극을 줄일 수 있었습니다.

### 4. 대시보드를 운영 질문 기준으로 재구성

대시보드는 많다고 좋은 게 아니라, 운영자가 바로 판단할 수 있어야 의미가 있습니다.

그래서 화면을 만들 때는 다음 질문을 기준으로 구성했습니다.

- 지금 어디가 비정상인가
- 최근 추세가 악화되고 있는가
- 메트릭 이상과 로그 이상이 같은 시점에 발생했는가
- 외부에서 접근 가능한 상태인가

이 기준으로 12+ 대시보드를 운영하면서, 단순 모니터링 화면이 아니라 운영 판단용 화면으로 정리했습니다.

## 구현 상세

이 프로젝트의 구현 핵심은 도구를 많이 붙이는 것이 아니라, 서로 다른 관측 데이터를 같은 운영 흐름으로 연결하는 데 있었습니다.

### 아키텍처 개요

- Prometheus가 50+ 메트릭 엔드포인트를 수집
- Blackbox Exporter가 외부 엔드포인트 상태를 감시
- Loki가 로그를 집계
- Elasticsearch가 추가 검색 및 분석 데이터소스로 연결
- Grafana가 3개 데이터소스를 하나의 시각화 레이어로 통합

### 구현 포인트 1. 메트릭 수집 계층

메트릭 계층에서는 수집 안정성과 조회 일관성을 우선했습니다.

운영자가 원하는 것은 수집기의 존재가 아니라, 상태 변화가 시간축으로 안정적으로 남는 것입니다. 그래서 엔드포인트를 늘리더라도 메트릭 명세와 대시보드 표현이 크게 흔들리지 않도록 구성했습니다.

### 구현 포인트 2. 로그와 메트릭의 교차 확인

로그는 단독으로 보면 노이즈가 많고, 메트릭은 단독으로 보면 이유가 부족할 때가 많습니다.

그래서 특정 이상 징후를 봤을 때 바로 로그 쪽 문맥으로 이어질 수 있게 구성했습니다. 이 방식은 장애 시점의 설명력을 높이는 데 효과적이었습니다.

### 구현 포인트 3. 운영자가 바로 쓰는 화면 설계

대시보드는 기술 데모가 아니라 운영 도구라고 보고 설계했습니다.

그래서 한 화면에 너무 많은 요소를 넣기보다, 목적에 따라 화면을 나눴습니다.

- 전체 상태 확인 화면
- 특정 지표 추세 확인 화면
- 로그 연계 확인 화면
- 외부 접근성 확인 화면

이렇게 나누면 대시보드 수가 늘어도 오히려 탐색 비용이 낮아집니다.

### 구현 포인트 4. Docker 기반 운영 단순화

구성 요소가 여러 개일수록 배포와 재기동 절차가 복잡해질 수 있습니다. Docker를 사용해 스택 구성과 관리 부담을 낮추고, 운영 환경 재현성을 확보했습니다.

## 결과

- 메트릭 수집 범위: 단일 관측 수준 → **50+ 메트릭 엔드포인트 실시간 수집**
- 시각화 범위: 단편적 확인 → **12+ Grafana 대시보드 운영**
- 데이터 연계: 개별 확인 → **Prometheus, Loki, Elasticsearch 3개 데이터소스 통합**
- 가용성 관리: 수동 추적 중심 → **99.5% 인프라 가용률 유지**

운영 측면에서도 의미가 있었습니다. 장애 징후를 하나의 화면에서 먼저 보고, 필요한 경우 메트릭과 로그를 이어서 확인하는 흐름이 만들어지면서 확인 동선이 짧아졌습니다.

또 이 플랫폼은 이후 다른 자동화나 운영 개선 작업을 붙일 수 있는 기반이 됐습니다. 관측이 정리되면 대응 자동화도 훨씬 설계하기 쉬워집니다.

## 사용 기술 상세

### Prometheus

메트릭 수집의 중심 역할을 담당했습니다. 수치 기반 상태 확인과 시계열 추세 분석에 활용했습니다.

### Blackbox Exporter

외부 엔드포인트 접근성과 응답 상태를 확인하는 용도로 사용했습니다. 내부 자원 상태와 실제 서비스 접근 가능성을 분리하지 않고 보기 위해 포함했습니다.

### Loki

로그 집계 레이어로 사용했습니다. 메트릭 이상 징후와 연결되는 로그 문맥을 빠르게 확인하는 데 유용했습니다.

### Elasticsearch

추가 검색과 분석이 필요한 데이터소스로 연동했습니다. Grafana 내부에서 다른 데이터와 함께 조회하는 구조를 만들었습니다.

### Grafana

운영자가 가장 자주 보는 접점입니다. 12+ 대시보드를 통해 전체 상태, 추세, 로그 연계, 외부 상태를 목적별로 나눠 시각화했습니다.

### Docker

구성 요소를 일관되게 운영하고 재현성 있는 배포 환경을 유지하기 위해 사용했습니다.
