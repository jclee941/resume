## 프로젝트 배경

증권 매매체결시스템은 장애나 보안 이슈가 곧바로 신뢰와 연결되는 영역입니다. 일반 서비스보다 더 높은 안정성과 통제 수준이 필요하고, 설계 단계와 운영 단계 모두에서 일관된 보안 기준이 유지돼야 합니다.

이 프로젝트는 2024.03부터 2026.02까지 이어진 금융 보안 인프라 프로젝트로, 구축 단계에서는 고가용성 중심의 보안 아키텍처를 설계하고, 운영 단계에서는 탐지와 알림, 정책 조회 자동화까지 확장한 점이 특징입니다. 즉, 한 번 만들고 끝나는 인프라가 아니라 실제 운영으로 이어지는 금융 보안 체계를 설계하고 다듬는 프로젝트였습니다.

클라이언트가 실제로 필요로 한 것은 단순 장비 도입이 아니었습니다. 본인가 심사 대응이 가능한 보안 구조, 장애를 견디는 가용성 설계, 반복 운영을 줄이는 자동화 체계, 그리고 실제 운영자가 계속 쓸 수 있는 탐지 및 대응 흐름이 함께 필요했습니다.

## 문제 상황

- 금융 시스템 특성상 높은 가용성과 통제 수준을 동시에 만족해야 했습니다.
- 구축 단계에서는 망 분리, 접근통제, 고가용성 설계를 함께 맞춰야 했습니다.
- 운영 단계에서는 보안 이벤트를 더 빠르게 보고, 더 짧은 경로로 대응해야 했습니다.
- 방화벽 정책 조회 같은 반복 업무가 운영 효율을 떨어뜨릴 수 있었습니다.
- 결과적으로 구축 단계와 운영 단계를 분리하지 않고, 하나의 보안 체계로 연결할 필요가 있었습니다.
- 공개 가능한 성과 기준으로는 **HA 99.99% 가용성 설계**, **금융위 본인가 심사 통과**, **32개 탐지 룰 운영**, **보안 이벤트 30초 내 알림**, **Ansible 정책 배포 표준화**, **방화벽 정책 조회 자동화**가 핵심이었습니다.

## 해결 접근

1. 구축 단계와 운영 단계를 별개 프로젝트처럼 보지 않고 연속된 보안 체계로 설계했습니다.
2. 구축 단계에서는 FGCP Active-Passive HA와 정책 배포 표준화를 중심으로 보안 인프라를 설계했습니다.
3. 운영 단계에서는 Splunk, n8n, FortiManager API를 활용해 탐지와 알림, 조회 자동화를 강화했습니다.
4. 금융 시스템에 맞는 통제 수준과 운영 효율을 동시에 확보하는 방향으로 기술을 선택했습니다.
5. 공개 가능한 범위와 NDA 범위를 명확히 나눠 기술했습니다.

### 1. 구축 단계, 보안 아키텍처와 컴플라이언스 기반 확보

구축 단계의 핵심은 장비 설치가 아니라, 본인가 심사와 실제 운영을 모두 견딜 수 있는 보안 구조를 만드는 것이었습니다.

주요 범위는 다음과 같았습니다.

- 고가용성 기반 증권 매매 시스템 보안 아키텍처 설계
- FGCP 기반 Active-Passive HA 구성
- 망 분리 및 접근통제 정책 설계
- 금융위원회 본인가 심사 기술 대응 및 보안 아키텍처 수립

여기서 중요한 점은, 설계 문서만 맞추는 것이 아니라 이후 운영 단계에서도 그대로 이어질 수 있는 구조를 만드는 것이었습니다.

### 2. 정책 배포를 사람 중심에서 표준화 중심으로 전환

금융권 보안 인프라는 일관성이 중요합니다. 초기 설정과 정책 배포가 사람마다 달라지면 운영 리스크가 커집니다.

그래서 Ansible Role을 활용해 방화벽 초기 설정과 정책 배포를 표준화했습니다. 이 선택은 단순 편의성이 아니라, 운영 품질을 일정하게 유지하기 위한 결정이었습니다.

### 3. 운영 단계, 탐지와 전달 속도 고도화

구축이 끝난 뒤에는 운영 속도가 중요해졌습니다. 보안 이벤트가 발생했을 때 사람이 늦게 보면 설계가 좋아도 운영 품질은 낮아질 수 있습니다.

그래서 Splunk 기반 탐지 체계를 운영하고, n8n 워크플로우와 연동해 실시간 알림 흐름을 구성했습니다. 이 과정에서 32개 탐지 룰 운영과 보안 이벤트 30초 내 알림이라는 운영 기준을 확보했습니다.

### 4. 조회 자동화로 운영 병목 해소

운영에서는 탐지만큼 조회 속도도 중요합니다. 필요한 정책을 확인할 때마다 수동 절차를 반복하면 분석과 판단보다 조회에 시간이 더 많이 들어갑니다.

FortiManager API를 이용한 방화벽 정책 자동 조회 툴을 개발해 이 병목을 줄였습니다. 이는 운영자의 시간을 더 중요한 보안 판단에 쓰게 해주는 변화였습니다.

## 구현 상세

이 프로젝트는 구축과 운영이 분리된 이력 나열이 아니라, 하나의 금융 보안 체계를 단계별로 고도화한 사례입니다.

### 1단계. 보안 인프라 구축, 2024.03 ~ 2025.02

#### 핵심 구현 내용

- 고가용성 기반 증권 매매 시스템 보안 아키텍처 설계
- FGCP(FortiGate Clustering Protocol) 기반 Active-Passive HA 구성
- Ansible Role을 활용한 보안 장비 초기 설정 및 정책 배포 표준화
- 망 분리 및 접근통제 정책 설계를 통한 보안 컴플라이언스 준수
- 금융위원회 본인가 심사 기술 대응 및 보안 아키텍처 수립

#### 왜 이런 방식이 필요했는가

금융 시스템에서는 보안이 강하면 운영이 불편해지고, 운영이 편하면 통제가 약해지는 식의 절충이 자주 발생합니다. 이 프로젝트에서는 둘 중 하나를 포기하지 않고, 설계 단계에서부터 일관성과 복구 가능성을 함께 고려했습니다.

FGCP Active-Passive HA는 가용성 확보의 중심이었고, Ansible Role 표준화는 같은 구성을 반복 가능하게 만들었습니다.

### 2단계. 보안 운영 고도화, 2025.03 ~ 2026.02

#### 핵심 구현 내용

- Splunk ES 및 FortiGate API를 활용한 보안 운영 자동화
- n8n 워크플로우를 연동하여 보안 이벤트 탐지 시 실시간 알림 연동
- FortiManager API를 이용한 방화벽 정책 자동 조회 툴 개발
- 32개 탐지 룰 운영
- 보안 취약점 점검 및 월간 정기 리포트 작성

#### 왜 이런 방식이 필요했는가

보안 인프라는 구축만 잘해도 충분하지 않습니다. 실제 운영에서는 이벤트를 보고, 분류하고, 필요한 정책을 확인하고, 반복 작업을 줄이는 체계가 필요합니다.

그래서 운영 단계에서는 Splunk, n8n, FortiManager API를 조합해 탐지, 전달, 조회를 연결했습니다.

### 구현 포인트 1. 구축과 운영의 기준 일치

설계 단계에서 만든 보안 구조가 운영 단계에서 유지되도록 하는 것이 중요했습니다. 따라서 구축 시점의 통제 기준과 운영 자동화 기준이 충돌하지 않게 설계했습니다.

### 구현 포인트 2. 가용성과 통제의 균형

HA 99.99% 가용성 설계는 단순 수치가 아니라 금융 환경에서 요구되는 안정성 목표를 반영한 결과였습니다. 동시에 망 분리와 접근통제 정책으로 통제 수준도 유지했습니다.

### 구현 포인트 3. 반복 업무 축소

정책 조회 자동화와 Ansible 기반 표준화는 모두 반복 업무를 줄이기 위한 수단이었습니다. 기술 선택의 기준은 복잡성이 아니라 운영 병목 해소였습니다.

### 구현 포인트 4. 실시간성 확보

운영 단계에서 30초 내 알림 체계를 만든 것은 단순 알림 도입이 아니라, 실제 대응 시간을 줄이는 데 목적이 있었습니다.

## 결과

- 가용성 설계: 일반 이중화 수준 → **HA 99.99% 가용성 설계**
- 컴플라이언스 대응: 준비 중심 → **금융위 본인가 심사 통과**
- 탐지 운영: 수동 확인 중심 → **32개 탐지 룰 운영**
- 알림 속도: 운영자 확인 의존 → **보안 이벤트 30초 내 알림 체계 구축**
- 정책 배포: 작업자별 편차 가능 → **Ansible 정책 배포 표준화**
- 정책 조회: 수동 장비 확인 중심 → **방화벽 정책 조회 자동화**

정량 성과 외에도 구축 단계와 운영 단계가 끊기지 않고 이어졌다는 점이 중요했습니다. 설계한 인프라가 실제 운영 자동화로 이어졌고, 운영 중 얻은 요구사항이 다시 보안 체계를 더 실용적으로 만들었습니다.

## 사용 기술 상세

### FortiGate, FortiManager

방화벽 계층의 중심 기술입니다. FGCP Active-Passive HA 구성과 정책 운영 체계의 핵심 축이었습니다.

### Splunk

운영 단계에서 보안 이벤트 탐지와 분석 중심 역할을 담당했습니다.

### Python

운영 자동화와 정책 조회 도구 개발에 활용했습니다. 반복 업무를 코드 자산으로 옮기는 데 적합했습니다.

### Ansible

초기 설정과 정책 배포 표준화에 활용했습니다. 금융 환경에서 중요한 일관성과 재현성을 확보하는 데 효과적이었습니다.

### n8n

운영 단계의 실시간 알림 흐름을 연결하는 자동화 레이어로 사용했습니다.

### Linux, VMware

운영 기반 환경으로 사용했습니다. 실제 금융 보안 인프라를 안정적으로 수용하는 운영 토대였습니다.

### FortiManager API

방화벽 정책 자동 조회 도구 개발의 핵심 인터페이스로 활용했습니다.
