## 프로젝트 배경

보안 운영에서 가장 시간이 많이 드는 작업 중 하나는 장비에 직접 접속해 필요한 정보를 찾고, 다시 사람이 해석 가능한 형태로 정리하는 일입니다. 정책 조회, 라우트 확인, 로그 통계 확인은 중요하지만 반복 빈도도 높아서 운영 효율을 빠르게 떨어뜨립니다.

특히 FortiManager와 FortiAnalyzer를 함께 쓰는 환경에서는 필요한 정보가 서로 다른 제품 API에 나뉘어 있습니다. 운영자는 같은 맥락의 질문을 풀기 위해 여러 시스템을 오가야 하고, 수동 확인 과정이 길수록 실수 가능성도 커집니다.

이 프로젝트는 그런 수동 반복을 줄이기 위해 시작했습니다. 목적은 단순한 API 래퍼 제작이 아니라, 방화벽 정책 조회, 라우트 관리, 로그 통계 확인처럼 자주 필요한 작업을 자동화 가능한 인터페이스로 정리하는 것이었습니다.

보안 운영에서는 조회 속도가 곧 판단 속도와 연결됩니다. 필요한 정보를 빨리 꺼내오지 못하면 분석보다 조회에 더 많은 시간이 들고, 반복 작업이 길어질수록 운영자는 더 피로해집니다. 이 라이브러리는 그 병목을 줄이는 데 초점을 맞췄습니다.

## 문제 상황

- 정책 조회, 라우트 관리, 로그 통계 확인이 장비별 수동 작업에 의존하고 있었습니다.
- FortiManager와 FortiAnalyzer를 각각 확인해야 해서 운영 흐름이 끊기기 쉬웠습니다.
- 자주 하는 조회 작업인데도 재사용 가능한 내부 도구가 부족했습니다.
- 수동 조회는 시간이 오래 걸리고 반복성이 높아 운영 집중도를 떨어뜨렸습니다.
- 결과적으로 FortiManager, FortiAnalyzer REST API를 하나의 자동화 관점으로 묶고, 수동 작업 대비 80% 시간 절감을 만드는 도구가 필요했습니다.

## 해결 접근

1. 사람이 자주 수행하는 보안 장비 조회 작업을 API 단위로 분해했습니다.
2. FortiManager와 FortiAnalyzer를 각각 직접 다루기보다 Python 라이브러리 레이어로 감쌌습니다.
3. 조회 중심 기능부터 정리해 자동화 파이프라인에 쉽게 붙일 수 있도록 설계했습니다.
4. 기능 추가보다 운영자가 실제로 반복하는 질문을 빠르게 해결하는 데 집중했습니다.
5. 추후 다른 자동화 도구나 보안 운영 흐름에 재사용할 수 있도록 인터페이스를 정리했습니다.

### 1. 수동 조회 절차를 작업 단위로 재정의

장비 자동화에서 중요한 것은 API를 많이 아는 것이 아니라, 사람이 어떤 질문을 얼마나 자주 던지는지 이해하는 것입니다.

이 프로젝트에서는 다음과 같은 실제 운영 질문을 중심으로 범위를 잡았습니다.

- 현재 정책을 빠르게 확인할 수 있는가
- 특정 라우트 상태를 관리할 수 있는가
- 로그 통계를 반복 조회하지 않고 가져올 수 있는가

### 2. Python 라이브러리로 재사용성 확보

일회성 스크립트는 당장은 빠르지만, 시간이 지나면 유지보수 포인트가 늘어납니다. 그래서 공통 호출 흐름을 Python 라이브러리로 정리해 다른 자동화 작업에서도 재사용할 수 있게 했습니다.

이 접근은 단순 조회 작업을 코드 자산으로 바꾸는 데 유리했습니다.

### 3. 제품별 API 차이를 사용성 관점에서 흡수

FortiManager와 FortiAnalyzer는 역할이 다르고 운영자가 얻고 싶은 결과도 다릅니다. 따라서 내부 구현은 제품별로 다르더라도, 사용하는 쪽에서는 작업 목적 중심으로 호출할 수 있게 만드는 것이 중요했습니다.

### 4. 운영 시간 절감이 기술 선택 기준

이 프로젝트에서 중요한 성과는 복잡한 기술 스택이 아니라, 수동 작업 대비 80% 시간 절감입니다. 그래서 추상화 방향도 멋진 구조보다 반복 시간을 줄이는 방향을 우선했습니다.

## 구현 상세

구현의 핵심은 FortiManager와 FortiAnalyzer를 단일 제품처럼 다루는 것이 아니라, 운영자가 필요한 결과를 빠르게 얻을 수 있게 공통 자동화 계층을 만드는 데 있었습니다.

### 아키텍처 개요

- Python이 자동화 라이브러리 레이어 역할 수행
- FortiManager REST API가 정책 조회와 라우트 관리 기능 담당
- FortiAnalyzer REST API가 로그 통계 수집 기능 담당
- 상위 자동화나 운영 스크립트가 이 라이브러리를 재사용하는 구조

### 구현 포인트 1. 조회 기능과 관리 기능의 경계 분리

정책 조회와 라우트 관리는 성격이 다릅니다. 조회는 빠르고 반복 가능해야 하고, 관리 기능은 호출 의도가 더 분명해야 합니다.

그래서 같은 제품을 다루더라도 기능 목적에 따라 인터페이스를 나누는 방향으로 설계했습니다.

### 구현 포인트 2. 로그 통계 수집 자동화

FortiAnalyzer 쪽은 운영자가 원하는 요약 결과를 빠르게 가져오는 쪽에 초점을 맞췄습니다. 모든 로그를 직접 보는 것이 아니라, 반복적으로 필요한 통계 확인을 자동화하는 것이 목적이었습니다.

### 구현 포인트 3. 운영 파이프라인 연결 용이성

라이브러리의 가치는 단독 실행보다 다른 도구와 붙을 때 커집니다. 이후 정책 조회 자동화나 보안 운영 흐름에 연결하기 쉽게 Python 중심으로 정리했습니다.

### 구현 포인트 4. 사람 중심 절차를 코드 자산으로 전환

이 프로젝트는 단순한 편의 도구가 아니라 운영자의 숙련도에 의존하던 절차를 코드로 옮기는 작업이었습니다. 그 결과, 반복 작업을 더 빠르고 일관되게 수행할 수 있는 기반이 생겼습니다.

### 구현 포인트 5. 보안 운영 도구의 재사용 기반 확보

한 번 만든 자동화가 다른 작업에도 이어질 수 있어야 투자 가치가 커집니다. 이 라이브러리는 개별 조회 스크립트가 흩어지는 상황을 막고, 이후 정책 조회 자동화나 운영 보조 도구에서 공통 기반으로 활용할 수 있게 정리했습니다.

## 결과

- API 적용 범위: 개별 제품 수동 확인 → **FortiManager, FortiAnalyzer REST API 통합 활용**
- 자동화 범위: 부분 스크립트 중심 → **정책 조회, 라우트 관리, 로그 통계 기능 정리**
- 운영 효율: 수동 장비 조회 중심 → **수동 작업 대비 80% 시간 절감**

정량 성과 외에도, 장비별 API를 따로 기억하지 않아도 된다는 점이 컸습니다. 운영자는 필요한 업무 목적에 집중하고, 구현 복잡도는 라이브러리 내부로 숨길 수 있었습니다.

또 이후 다른 보안 자동화 프로젝트에서 재사용 가능한 기반을 확보했다는 점도 실무적으로 의미가 있었습니다.

즉, 이 프로젝트의 결과는 단순한 스크립트 하나가 아니라 보안 장비 자동화를 위한 공통 인터페이스를 만든 데 있습니다.

## 사용 기술 상세

### Python

자동화 라이브러리의 구현 언어로 사용했습니다. 재사용성과 유지보수성을 고려해 공통 호출 레이어를 정리하는 데 적합했습니다.

### FortiManager REST API

방화벽 정책 조회와 라우트 관리 작업에 활용했습니다. 운영자가 자주 확인하는 관리성 작업을 자동화 관점으로 가져오는 데 핵심 역할을 했습니다.

### FortiAnalyzer REST API

로그 통계 수집 기능에 활용했습니다. 보안 운영에서 반복되는 로그 확인 절차를 더 짧게 만드는 데 기여했습니다.
