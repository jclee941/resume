# ADR 0007: MSA Service Split — Independent Workers for Portfolio and Job Dashboard

- Status: Accepted
- Date: 2026-03-22
- Supersedes: [ADR 0006](0006-single-worker-architecture.md)

## Context

ADR 0006 established a single-worker architecture where portfolio and job-dashboard shared one Cloudflare Worker entry point (`entry.js`). As the job-dashboard grew to 47 API endpoints, 14 handlers, 7 Cloudflare Workflows, a Durable Object, and a Queue consumer, this created a single failure domain, entangled deployment lifecycles, and made independent scaling impossible.

The MSA refactoring (Phases 0–3) split the system into independently deployable Cloudflare Workers connected via Service Bindings.

## Decision

The project adopts a microservice architecture with three independent services:

### Service Boundaries

| Service              | Worker Name   | Domain                  | Responsibility                                                   |
| -------------------- | ------------- | ----------------------- | ---------------------------------------------------------------- |
| Portfolio Worker     | `resume`      | `resume.jclee.me`       | Edge static rendering, HTML templates, i18n, SEO, health/metrics |
| Job Dashboard Worker | `job`         | `resume.jclee.me/job/*` | 47 API endpoints, 7 CF Workflows, Queue consumer, Durable Object |
| Job Server           | N/A (Node.js) | Docker container        | MCP server, 10 platform crawlers, Fastify runtime                |

### Communication

Portfolio proxies `/job/*` requests to the job-dashboard worker via Cloudflare Service Binding (`env.JOB_SERVICE.fetch(request)`). This is a same-datacenter RPC call with no network overhead.

### Shared State: KV Namespaces

Both the portfolio worker and job-dashboard worker bind to the same KV namespace IDs for `SESSIONS`, `RATE_LIMIT_KV`, and `NONCE_KV`. This is **intentional shared state**, analogous to two microservices sharing a database:

- **SESSIONS**: Auth session tokens must be readable by both workers since the job-dashboard validates auth on incoming requests, and sessions are created through the portfolio's proxy path.
- **RATE_LIMIT_KV**: Rate limiting is enforced per-IP across the entire domain; splitting would allow double the requests by hitting both workers.
- **NONCE_KV**: CSRF nonces generated by one worker must be validatable by the other during cross-worker request flows.

Splitting these into per-worker namespaces would break auth continuity and rate limiting. The shared KV layer is an explicit, documented coupling point — not an accidental violation of service independence.

### Shared Code: `@resume/shared`

Cross-cutting concerns (Elasticsearch client, Logger, error types, user-agent parsing, phone formatting, job categories, browser automation, Wanted API client) live in `packages/shared/` and are imported by both workers. ESLint boundary rules (`no-restricted-imports`) enforce that workers never import from each other — only from `@resume/shared/*`.

### CI/CD Independence

- **Portfolio**: Has a custom build pipeline (`generate-worker.js` → `worker.js`). CI validates this build in the `build` job. Production deploy via Cloudflare Workers Builds.
- **Job Dashboard**: Standard wrangler project deployed from source. CI validates the wrangler configuration in the `validate-cloudflare` job (wrangler types generation). Production deploy via Cloudflare Workers Builds independently.
- **Change detection**: `tools/ci/affected.go` independently detects changes to each worker. Changes to `packages/shared/` trigger validation of both workers.

### Deployment Authority

Cloudflare Workers Builds is the sole production deploy authority for both workers. GitHub Actions CI is validation-only — it runs lint, typecheck, tests, security scans, and build validation but never deploys.

## Consequences

- Positive: Independent failure domains — a bug in job-dashboard does not crash the portfolio.
- Positive: Independent deployment lifecycles — each worker can be deployed without affecting the other.
- Positive: Cleaner code organization — 7 route modules, shared kernel, ESLint-enforced boundaries.
- Positive: Independent scaling — each worker scales to its own traffic pattern.
- Negative: Shared KV namespaces create a coupling point that requires coordination for KV schema changes.
- Negative: Service Binding adds a proxy hop for `/job/*` requests (negligible latency within same datacenter).
- Follow-up: Phase 4 (observability) will add per-service Elasticsearch indices and Grafana dashboards (deferred, not blocking).
